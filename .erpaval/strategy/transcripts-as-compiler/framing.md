# Framing — transcripts-as-compiler

**Status:** COMPLETE

## Challenge

Should we spend the next quarter (90-day bet) lifting `claude-sql` from a single-developer transcript query tool into the substrate for **transcripts-as-PR-review** — where every merged pull request ships with the prompt→agent→diff transcript that produced it, and that transcript becomes a first-class, searchable, attributable artifact alongside the diff? The short-term product decision is concrete: invest the quarter in (a) a binding from PR → transcript, (b) a reviewer-facing compression of the transcript into a review-sized artifact, and (c) a minimum team-corpus ingestion path, or do not. The long-term thesis this decision is a down-payment on: **source code is the output of a compile step whose real source is the transcript, and today we throw the source away.**

## Audience

- **Primary decision-maker:** Laith Al-Saadoon (Principal AI Engineer, AWS AGS Tech AI Engineering NAMER). Reads diffs fluently, ships production AI on Bedrock, already owns the substrate. Does not need an embedding 101; does need a clean build-vs-buy read and a named crux.
- **Likely co-investors / near-peers:** AGS Tech AI Engineering NAMER teammates and adjacent AWS platform/DevEx groups who care about agentic developer productivity, code-review throughput, and "AI-authored code" governance.
- **What they already agree on:**
  - Agents write production code now; that's table stakes (`CLAUDE.md` "What this project is" enumerates the existing pipeline).
  - PR-based code review is strained by AI-authored diffs — reviewers read LLM output without the context that produced it.
  - Transcripts exist and are cheap to capture; `claude-sql` already proves you can query them at rest with SQL + embeddings + LLM classification.
- **Where this thesis will get pushback:**
  1. **Noise.** Transcripts are 10–100× the token volume of the diff; reviewers will ignore them by default unless the intelligence layer earns its keep.
  2. **Privacy/IP.** Transcripts contain half-formed ideas, rejected paths, secrets pasted into prompts, and third-party content — pushing them into a shared review surface is a governance problem, not just an engineering one.
  3. **Binding fragility.** Git has no native "which transcript produced this commit?" primitive; any binding we invent is out-of-band until a host (GitHub, Bedrock AgentCore, a git-notes convention) adopts it.
  4. **Distribution.** GitHub, Cursor, and the agent vendors themselves can eat this if it proves valuable — a standalone tool has to be either (a) so substrate-shaped that the incumbents integrate with it, or (b) so vertical that it ships inside an org before the incumbents notice.

## Crux read (candidate)

**The crux is binding — attaching a specific transcript to a specific merged PR in a way that survives rebase, squash-merge, multi-author commits, and cross-tool handoffs.** Without a stable PR↔transcript binding, every downstream capability (review compression, replay, forensics, team analytics) operates on a disconnected corpus and collapses to "logs we happen to have." Noise, governance, and substrate-lift are all real, but they are solvable *inside* a binding that exists; they are unsolvable *without* one.

Reason for choosing binding over the three alternates:

- *Signal-to-noise* is a compression problem the existing `claude-sql` intelligence layer (Sonnet 4.6 classification, friction detection, cluster/community analytics — `CLAUDE.md` "What this project is", lines referenced throughout) already mostly solves; pointing it at a bound corpus is a product decision, not a research one.
- *Trust/governance* is a policy + redaction problem — hard, but orthogonal to whether the artifact exists. An unbound transcript has the same IP/PII risks as a bound one; binding does not make governance harder.
- *Substrate-lift* (multi-user ingestion, ACLs, retention) is a build cost, not a conceptual bottleneck. It unblocks *scale*, not *whether the thing works*. Binding is what decides whether the thing works.

The rumelt-architect will sharpen or overrule this in Phase 2; flagging binding as the candidate crux so Wardley and Minto can be composed around it.

## Stakes

- **If the thesis is right:** `claude-sql` (or its v2 successor) becomes **the git-blame for AI-authored code** — a load-bearing artifact in every PR review, every incident retro, every onboarding walkthrough. Reviewers stop reviewing just the diff and start reviewing the prompt→decision→diff path. Transcripts become the primary source-of-record for *why* software changed; diffs become the compiled output. AWS owns the reference substrate for this on Bedrock (global CRIS Cohere Embed v4 + Sonnet 4.6 — `CLAUDE.md` Bedrock section), giving AGS a durable narrative beat into GitHub/Cursor/vendor-owned closed surfaces.
- **If the thesis is wrong (or we're too slow):** `claude-sql` stays a useful personal analytics tool — genuinely valuable for Laith's own workflow, but strategically unremarkable — and the "transcript-as-artifact" thesis gets claimed by whoever has distribution: GitHub Copilot Review (already shipping LLM review feedback without transcript provenance — nova_web_grounding 2026-05-08, https://docs.github.com/en/copilot/using-github-copilot/code-review/using-copilot-code-review), Cursor via SpecStory / cursor-replay / vibe-replay-style extensions (early signal, nothing PR-bound yet — nova_web_grounding 2026-05-08, https://forum.cursor.com/t/built-a-cursor-extension-to-save-and-share-chat-and-composer-history/35314, https://vibe-replay.com/), or Anthropic itself shipping a native Claude Code PR-replay feature.
- **Option value of not betting:** the intelligence layer `claude-sql` already ships (18 DuckDB views, Cohere Embed v4 on global CRIS, Sonnet 4.6 structured output, UMAP+HDBSCAN, Louvain, friction classifier, persistent HNSW, sharded parquet caches — `CLAUDE.md` "What this project is" + "Resilience patterns to preserve") continues to earn its keep as a personal tool and as a reference implementation for Bedrock-on-agent-analytics. The downside of betting wrong is bounded; the downside of not betting is ceding a category.

## Known facts

Grounded in code, docs, and one round of external web search. Every fact below is cited inline to either a file path or a URL.

1. **`claude-sql` already ships the analytics pipeline.** 18 DuckDB views + 14 macros normalize messages, tool calls, todos, subagents, and costs; Cohere Embed v4 on `global.cohere.embed-v4:0` CRIS produces embeddings; DuckDB VSS HNSW serves top-k cosine search; Sonnet 4.6 on `global.anthropic.claude-sonnet-4-6` CRIS does session classification, per-message trajectory, stance conflicts, and user-friction labeling with Bedrock GA `output_config.format` structured output; UMAP+HDBSCAN clusters messages and c-TF-IDF names clusters; Louvain over session centroids names communities. (`CLAUDE.md` "What this project is" + "Bedrock" + "Structured output (Sonnet classification)" sections; `src/claude_sql/` contains `sql_views.py`, `embed_worker.py`, `llm_worker.py`, `cluster_worker.py`, `community_worker.py`, `friction_worker.py`, `judge_worker.py`, `terms_worker.py`, `ungrounded_worker.py`, `schemas.py`.)
2. **Corpus is personal, JSONL-on-disk, ~2GB / 15K files on Laith's machine.** `~/.claude/projects/**/*.jsonl` plus subagent sidecars under `~/.claude/projects/*/*/subagents/agent-*.jsonl`. Read zero-copy by DuckDB with `read_json(..., filename=true, union_by_name=true, sample_size=-1, ignore_errors=true, maximum_object_size=67108864)`. No multi-user story exists today. (`CLAUDE.md` "What this project is" + "Resilience patterns to preserve" + "Deferred decisions" Snapshot-tier entry; `README.md` "How it works".)
3. **Resilience patterns are already load-bearing.** Persistent HNSW at `~/.claude/hnsw.duckdb` (lazy-built, mtime-checked, self-heals on corruption); sharded parquet caches under `<cache>/part-<ts_ns>.parquet` (workers append, readers glob); per-session checkpointers + per-uuid anti-join make reruns free; tenacity retries wrap Bedrock with exponential backoff; DuckDB tuned via `CLAUDE_SQL_DUCKDB_*` env vars (memory 70% of host, temp under `~/.claude/duckdb_tmp`); friction worker has a zero-cost regex fast path + Sonnet fallback (`CLAUDE.md` "Resilience patterns to preserve"; `src/claude_sql/checkpointer.py`, `src/claude_sql/parquet_shards.py`, `src/claude_sql/retry_queue.py`).
4. **Cost guard is already built.** Every subcommand that spends Bedrock money defaults to `--dry-run`; dry-run path uses pure-SQL `COUNT(DISTINCT session_id)` to stay fast on large corpora. `claude-sql analyze` chains embed → cluster → classify → trajectory → conflicts → friction and respects `--dry-run` per step. (`CLAUDE.md` "Bedrock" + "Friction classifier" + "Resilience patterns to preserve"; `src/claude_sql/cli.py`, `src/claude_sql/llm_worker.py`.)

8. **Competitive context (brief external grounding, 2026-05-08).**
   - GitHub Copilot Code Review ships LLM-generated PR comments tied to specific lines/commits, but **does not emit transcript provenance / audit trail** as a public artifact. Reviews consume a premium request and are advisory, not blocking. (nova_web_grounding 2026-05-08; https://docs.github.com/en/copilot/using-github-copilot/code-review/using-copilot-code-review)
   - **Cursor has no first-party session replay bound to PRs.** Third-party extensions fill the gap: SpecStory exports chat/composer history to markdown with shareable web links; cursor-replay and vibe-replay convert sessions into self-contained HTML files embedding prompts + diffs + tool calls. Early-stage, developer-targeted, not PR-bound. (nova_web_grounding 2026-05-08; https://forum.cursor.com/t/built-a-cursor-extension-to-save-and-share-chat-and-composer-history/35314, https://vibe-replay.com/)

   - Net read: "transcript-as-first-class-PR-artifact with stable binding" is **unclaimed territory** as of the research date. The incumbents have the distribution; they have not built the artifact. Window exists; it is not permanent.

## Open questions

Named so Phase 2 doesn't re-derive them.

- **PR ↔ transcript binding mechanism.** Candidates, each with real tradeoffs: (a) git commit trailer like `Claude-Transcript-Id: <uuid>` — simple, survives cherry-pick, dies on squash-merge unless the squashed trailer is preserved; (b) `git notes` — designed for exactly this but near-invisible in GitHub UI and non-default to push; (c) out-of-band index keyed on `(repo, branch, commit-range, author, timestamp)` — robust to VCS ops but fragile across rebases; (d) code-host-native annotation (GitHub PR comment, GitLab note, Bedrock AgentCore artifact) — high adoption ceiling but vendor-locked. Which of these is the 90-day bet?
- **Artifact locality.** Is "every PR ships with its transcript" a *git-level primitive* (transcript lives in-tree as `.transcripts/<commit>.jsonl`, bloats the repo, breaks partial-clone workflows), a *code-host feature* (a sidebar panel fetched from an adjacent service, Codecov-style), or a *separate companion tool* (install-optional, opt-in, like pre-commit)?
- **Viewer audience and granularity.** Who sees the transcript — PR author only (self-review tool), assigned reviewers (Codecov-scoped), full repo members (open-source norm), or org-wide (governance-driven)? Does the reviewer see the raw transcript, a compressed LLM summary, or both behind a toggle? Default matters more than the full menu.
- **Adoption model.** Three viable shapes: (a) **individual opt-in** à la Copilot — each dev decides per-commit whether to attach; (b) **team opt-in** à la Codecov — repo-level config, applies to every PR once enabled; (c) **org mandate** à la SOC2 — governance-required for AI-authored code across compliant orgs. Each has different build priorities (a = zero-friction CLI, b = web app + GitHub App, c = policy engine + redaction).
- **Substrate identity.** Is `claude-sql` the v2 substrate itself — lifted to multi-user with auth, ACLs, retention — or a reference implementation whose code-level patterns (view schema, HNSW index, structured-output schemas, friction classifier) inform a *different* v2 product? The first preserves velocity; the second reduces personal-tool / team-tool coupling risk.
- **Governance and redaction surface.** Transcripts contain secrets users paste into prompts, rejected paths, third-party IP, and inter-agent communications. What's the minimum viable redaction pipeline that makes a transcript safe for the viewer audience above — prompt-injection scrub, regex/secret scanners, org-policy pre-ingest filter, per-transcript ACL? (Out of scope for the 90-day bet on whether to bet — but in scope for coherent actions if Rumelt commits.)
- **Deferred-item ordering under the team-corpus lift.** Snapshot tier / CacheNode DAG probably gets forced (re-scanning team JSONL on every query is untenable past ~10GB). Batch embeddings via `CreateModelInvocationJob` probably gets forced by team-corpus volume. Do these become the tactical backlog for the substrate-lift sub-bet, or do they wait for the first real bottleneck?
